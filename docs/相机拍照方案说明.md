# HarmonyOS 外卖送达凭证拍照方案

## 1. 业务背景

外卖员送达订单时需要连续拍摄多张送达凭证。目标交互是：

1. 进入一次拍摄页面。
2. 拍完一张后自动保存，不进入单张确认页面。
3. 保持相机预览，可以继续拍摄下一张。
4. 显示已拍照片数量。
5. 最后统一确认并关联订单或批量上传。

系统相机适合单张拍摄，但系统控制的“拍照—确认—返回”流程不能由应用取消。因此，该业务应使用应用内自定义相机。

## 2. 系统相机与自定义相机的区别

| 对比项 | 系统相机（CameraPicker） | 自定义相机（Camera Kit） |
| --- | --- | --- |
| 拍摄界面 | 系统提供，应用不能修改 | 应用使用 `XComponent` 自行实现 |
| 相机权限 | 通常不需要应用直接申请 | 必须声明并动态申请 `ohos.permission.CAMERA` |
| 相机资源 | 由系统相机管理 | 由应用管理 |
| 返回结果 | 一次返回一个 `PickerResult` | 每次拍摄通过回调获得一个 `PhotoAsset` |
| 单张确认页面 | 由系统控制，不能取消 | 完全由应用决定 |
| 连续拍摄 | 不适合，每次通常都要确认并返回 | 适合，可复用同一相机会话连续拍摄 |
| 对焦、变焦、闪光灯 | 主要由系统相机处理 | 应用可以按设备能力自行控制 |
| 前后台生命周期 | 系统负责相机资源 | 应用需要释放并恢复相机资源 |
| 开发成本 | 较低 | 较高 |
| 典型场景 | 头像、单张报销凭证 | 连拍、扫码、证件扫描、送达凭证 |

### 2.1 调用系统相机

系统相机通过 `CameraPicker` 拉起：

```ts
import { camera, cameraPicker } from '@kit.CameraKit';

const profile: cameraPicker.PickerProfile = {
  cameraPosition: camera.CameraPosition.CAMERA_POSITION_BACK
};

const result: cameraPicker.PickerResult = await cameraPicker.pick(
  getContext(),
  [cameraPicker.PickerMediaType.PHOTO],
  profile
);

const photoUri: string = result.resultUri;
```

调用流程：

```text
应用 → 拉起系统相机 → 用户拍照 → 系统确认/重拍 → 返回一张照片 URI
```

再次拍照需要再次调用 `cameraPicker.pick()`，因此无法实现保持当前预览、连续拍摄多张后统一确认。

### 2.2 使用自定义相机

自定义相机不使用 `CameraPicker`，而是直接使用 Camera Kit：

```ts
import { camera } from '@kit.CameraKit';
```

主要对象包括：

- `CameraManager`
- `CameraDevice`
- `CameraInput`
- `PreviewOutput`
- `PhotoOutput`
- `PhotoSession`
- `XComponent`

初始化流程：

```text
申请相机权限
  → 获取 CameraManager 和 CameraDevice
  → 创建 CameraInput
  → 创建 PreviewOutput 和 PhotoOutput
  → 创建 PhotoSession
  → 添加输入流和输出流
  → 提交并启动会话
```

相机会话只初始化一次，之后每次点击快门调用：

```ts
await photoOutput.capture(photoSetting);
```

拍照结束后不关闭 `PhotoSession`，所以预览可以保持运行并继续拍摄。照片通过以下事件返回：

```ts
photoOutput.on('photoAssetAvailable', (error, photoAsset) => {
  // 保存照片，并将 URI 加入当前订单的送达凭证数组。
});
```

## 3. 连续拍照实现原则

这里的“连拍”是连续点击快门拍摄多张，而不是相机硬件的长按高速连拍模式。

实现时应遵循：

1. 预览流、拍照流和相机会话只初始化一次。
2. 每次点击只调用一次 `PhotoOutput.capture()`。
3. 收到 `photoAssetAvailable` 后自动保存照片。
4. 保存完成后不退出页面、不关闭相机会话。
5. 使用 URI 数组记录本次订单的所有照片。
6. 最后点击“完成”时统一回传订单页或批量上传。
7. 使用 `isCapturing` 防止同一时刻重复提交拍摄请求。
8. 同时统计待保存照片，避免异步回调导致数量超过上限。

当前示例最多允许拍摄 9 张：

```ts
const MAX_PHOTO_COUNT: number = 9;
```

## 4. 前后台切换与相机恢复

### 4.1 只使用 CameraPicker

如果应用全程只调用系统 `CameraPicker`，应用没有持有相机输入流、输出流或会话，因此不需要自行重启这些资源，只需要处理返回的 `PickerResult`。

### 4.2 使用自定义相机

如果应用持有自定义相机，进入后台或拉起系统相机后，相机流可能因安全策略被强制断开。再次进入应用时，不应直接复用旧的 `CameraInput`、`PreviewOutput`、`PhotoOutput` 和 `PhotoSession`。

推荐流程：

```text
自定义相机运行
  → 应用进入后台或拉起系统相机
  → 停止并释放自定义相机
  → 应用恢复前台
  → 重新创建输入流、预览流、拍照流和相机会话
  → 恢复预览
```

页面隐藏时释放：

```ts
async onPageHide(): Promise<void> {
  this.cameraReady = false;
  await cameraService.release();
}
```

页面重新显示时恢复：

```ts
async onPageShow(): Promise<void> {
  if (this.surfaceId !== '' && !this.cameraReady) {
    await cameraService.start(this.context, this.surfaceId);
    this.cameraReady = true;
  }
}
```

如果在 `cameraPicker.pick()` 返回处恢复相机，同时又在 `onPageShow()` 中恢复，需要使用 `cameraReady`、`isInitializing` 等状态防止重复初始化，否则可能出现设备忙、黑屏、重复添加输出流或会话提交失败。

## 5. 模拟器支持情况

较新版本的 DevEco Studio 本地模拟器支持 Camera Kit 的基础预览和拍照，图像来自电脑摄像头，可以验证本项目的连续拍照交互。

使用模拟器时需要：

1. 使用 Phone 本地模拟器，不使用云模拟器。
2. 电脑具有可用摄像头。
3. 允许 DevEco Studio 或模拟器访问电脑摄像头。
4. 关闭微信、会议软件等可能占用摄像头的程序。
5. 首次启动应用时允许相机权限。

模拟器适合验证：

- 相机权限流程。
- 基础预览和拍照。
- 连续点击拍照。
- 照片数量、上限和统一完成逻辑。
- 页面前后台切换时的资源释放和恢复。

模拟器不能准确验证：

- 多摄像头切换。
- 闪光灯、广角、长焦等硬件能力。
- 真实成像质量和厂商算法。
- 真机性能、功耗和稳定性。

如果 `getSupportedCameras()` 返回空数组或相机启动失败，应检查模拟器版本、电脑摄像头权限及摄像头占用情况。

## 6. 当前项目实现位置

### 相机服务

文件：`entry/src/main/ets/camera/ContinuousCamera.ets`

负责：

- 获取相机设备能力。
- 创建 `CameraInput`、`PreviewOutput` 和 `PhotoOutput`。
- 创建并启动 `PhotoSession`。
- 调用 `PhotoOutput.capture()` 拍照。
- 监听 `photoAssetAvailable`。
- 保存照片并返回 URI。
- 释放全部相机资源。

### 拍摄页面

文件：`entry/src/main/ets/pages/Index.ets`

负责：

- 显示 `XComponent` 相机预览。
- 控制快门按钮状态。
- 记录照片 URI 数组。
- 显示已拍数量和最多 9 张的限制。
- 在页面隐藏时释放相机。
- 在页面重新显示时恢复相机。
- 点击“完成”后统一处理本次拍摄结果。

### 权限申请

文件：`entry/src/main/ets/entryability/EntryAbility.ets`

负责动态申请 `ohos.permission.CAMERA`。

文件：`entry/src/main/module.json5`

负责声明相机权限及使用原因。

## 7. 生产环境建议

当前示例将照片保存后记录到 `photoUris`。接入真实订单业务时，建议：

1. 为每张照片记录订单 ID、拍摄时间和上传状态。
2. 在照片保存后立即异步上传，降低最后统一上传的等待时间。
3. 点击“完成”前确认所有待保存任务已经结束。
4. 对上传失败的照片提供单张重试能力。
5. 服务端校验照片数量、文件类型和大小。
6. 页面退出时提示尚未提交的照片，避免凭证丢失。
7. 根据隐私要求决定保存到系统相册还是应用沙箱；送达凭证通常更适合应用沙箱和业务服务器。
